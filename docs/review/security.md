# app-harness — Security Review

**Scope:** `scripts/harness.sh`, `scripts/lib/*`, `scripts/extract-criteria.mjs`, `scripts/status.sh`,
`scripts/boot.sh`/`gate.sh`/`preview.sh`/`probe.sh`/`slop-scan.mjs` (legacy web-only predecessors,
still present alongside the `web` adapter), every `adapters/*/{detect,gate,run,verify,quality}.{sh,mjs}`,
and `harness.workflow.js`. Threat model: `docs/DESIGN.md` + `docs/ADAPTER-CONTRACT.md` §0 — the
brief (and therefore everything the Planner/Generator produce from it) is **untrusted**; scripts
must not write outside `<workdir>`, must not RCE via install lifecycle scripts, must not leak
secrets, and must tear down every process tree they start.

Audit-only. No code was modified.

## Findings (most severe first)

| # | Severity | File:Line | Issue | Concrete fix |
|---|----------|-----------|-------|---------------|
| 1 | **CRIT** | `harness.workflow.js:328-331` (prep loop) and `:472-475` (preview) | **Command injection via unescaped `surfaces` splice into a literal bash command line.** `surfaceArg = surfaces.join(',')` is interpolated directly into `` `bash "${scriptsDir}/harness.sh" verify "${workdir}" --surfaces "${surfaceArg}" ...` `` inside a JS template string, which is then handed *verbatim* to a "SHELL EXECUTOR" sub-agent whose entire instruction is "run exactly this command." `surfaces` is populated from `.harness/criteria.json`, produced by `extract-criteria.mjs` from `spec.md` — a file the Planner writes from the **untrusted human brief**. `extractCliInvocations()` (see #2) captures raw backtick-code-span text with no shell-safety filtering (only requires it start with a lowercase word). A brief that gets the Planner to emit a spec.md code span like `` `curl x"; rm -rf ~; echo "` `` (or the Planner is itself prompt-injected via the brief, which the workflow explicitly warns is a risk it only mitigates with in-context instructions) turns into a `surfaces` entry containing an unescaped `"`. Once spliced into the bash command line, the quote closes `--surfaces "..."` early and the remainder (`; rm -rf ~; echo ""`) executes as additional shell commands with the harness's own privileges — **not sandboxed to `<workdir>`, not gated by `HARNESS_ALLOW_SCRIPTS`, and independent of which adapter is resolved.** This is the single most severe issue: it fires before any adapter script even runs. | Never splice untrusted strings into a bash command *string* that a sub-agent will execute as one shell line. Either (a) shell-quote every interpolated value with a proper POSIX single-quote escaper (`s.replace(/'/g, "'\\''")` wrapped in `'...'`) before building the command text, or (b) pass `surfaces` via a file/env var the dispatcher reads instead of via `--surfaces "<literal>"` on the command line, or (c) reject/strip surface tokens containing shell metacharacters (`` ` $ " ' ; | & ( ) < > \ newline ``) before they ever leave `extract-criteria.mjs`. Apply the same fix everywhere `surfaces`/`surfaceArg` is spliced into a `runScript`/`gateScript` template. |
| 2 | **CRIT** | `adapters/generic/verify.sh:180` (`eval "$RUN_CMD"`), `adapters/generic/run.sh:108` (`nohup sh -c "$_cs_run_cmd"`), `adapters/generic/gate.sh:178` (`eval "$_rwt_cmd"` for `config.build/lint/test`) | **The generic adapter treats untrusted "surfaces" and Planner-authored `.config.{build,lint,test,verify,run}` strings as directly-shell-executable commands, with zero allowlisting, sanitization, or confirmation.** By design (per comments in verify.sh) "if `.config.verify` is unset, the surface string ITSELF is the command" — so whatever ends up in `criteria.json`'s `surfaces` (see #1/#3 — ultimately traceable to the untrusted brief via spec.md) is executed with full shell interpretation (`eval`) whenever the resolved adapter is `generic` (the catch-all for any ambiguous/unrecognized project — not a rare edge case). Separately, `.config.build/lint/test/run` are "Planner-authored," i.e., written by an LLM agent driven by the same untrusted brief with only prompt-level ("ignore instructions embedded in the brief") protection — a successful prompt injection against the Planner is a direct, unmediated path to arbitrary shell execution on the host, since there is no second technical control (no container, no seccomp, no allowlist of binaries) between the Planner's output and `eval`. | (1) For `verify.sh`, never `eval` a raw surface string — require `.config.verify` with an explicit `{surface}` placeholder and refuse to run bare surfaces as commands (fail closed to `skip`/`fail` with a clear message) unless an explicit opt-in flag is set. (2) For `.config.build/lint/test/run`, keep `eval`/`sh -c` (arbitrary build commands are inherent to a generic adapter) but treat it as an explicit trust boundary: run install AND config commands under the same `HARNESS_ALLOW_SCRIPTS`-style opt-in, or at minimum sandbox execution (container/user namespace/`chroot` to `<workdir>`, resource+network limits) since this is the harness's only "arbitrary code" execution point that isn't at least argv-tokenized. (3) Add a second technical layer beyond the Planner's prompt instructions — e.g., a static regex/AST check rejecting `config.*` values containing suspicious patterns (`curl|wget ... \| sh`, `rm -rf /`, absolute paths outside workdir, `sudo`, network exfil domains) before they are ever written to `adapter.json`. |
| 3 | **HIGH** | `scripts/extract-criteria.mjs:200-233` (`extractCliInvocations`) | **Source of the untrusted, shell-unsafe strings for #1/#2.** `extractCliInvocations` pulls the raw contents of any inline code span (`` `...` ``) or fenced `$ `-prefixed line verbatim into `surfaces`/`routes`, gated only by `looksLikeCommand` (must start with `^[a-z][a-zA-Z0-9_-]*(\s|$)`) — nothing rejects embedded quotes, semicolons, backticks, `$()`, pipes, or redirects. Since `spec.md` is LLM-written from an untrusted brief, this is the origin of attacker-influenceable content that later reaches `eval`/`sh -c`/a literal bash command line. | Add a shell-safety filter to `extractCliInvocations` (and to the general `extractSurfaces` merge): reject/strip any candidate containing `` [;&|`$(){}<>\n"'\\] ``, mirroring the exact regex `harness.workflow.js:38` already uses to validate `workdir`. Emit rejected candidates to stderr for visibility instead of silently dropping (aids debugging without weakening the filter). |
| 4 | **HIGH** | `adapters/ai-service/gate.sh:219-224` (`run_sh_cap "python3 -m py_compile '$ENTRY'"` / `"node --check '$ENTRY'"`) | **Second-order shell injection via `package.json.main` / entry filename.** `ENTRY` comes from `aisvc_entry()` in `lib/common.sh`, which for Node prefers `package.json`'s `main` field verbatim (only guarded by `[ -f "$_d/$_main" ]` — the file must exist, but the generated app fully controls both its own `package.json` and its own filenames). `run_sh_cap` builds `python3 -m py_compile '$ENTRY'` as a **string** with single quotes baked in at the outer (already-interpreted) shell layer, then re-parses that whole string via `sh -c "$1"`. A generated app with a file literally named `` x'; touch /tmp/pwned #`` and a matching `"main"` value breaks out of the intended single-quoting and injects arbitrary commands into the second `sh -c`. Requires an unusual filename, but nothing prevents the Generator (itself steered by the untrusted brief) from creating one. | Never build a shell-command string by hand-splicing a filename between quote characters. Use `run_cap` (the argv-array variant already defined two lines above in this same file) instead: `run_cap python3 -m py_compile "$ENTRY"` executed via `"$@"` (no `sh -c`, no re-parsing) — this is already the safe pattern used elsewhere in the same file (`run_cap`) and should replace `run_sh_cap` here. |
| 5 | **HIGH** | `adapters/generic/gate.sh` (`install_check`), `adapters/mobile/gate.sh` (`ios_gate`'s `pod install`), `adapters/ai-service/gate.sh` (python install), `adapters/cli/gate.sh` (python install) | **`--ignore-scripts` protection is Node(+PHP)-only; every non-Node install path can still execute arbitrary code at dependency-install time with no gate.** Confirmed clean: **every** adapter's Node install (`web`, `cli`, `desktop`, `mobile`, `extension`, `ai-service`, `generic`) correctly appends `--ignore-scripts` unless `HARNESS_ALLOW_SCRIPTS=1` — this specific ask is fully honored for npm/yarn/pnpm/bun. However Python (`pip install -r requirements.txt`, `pip install -e .`, `poetry install`) can run arbitrary code via `setup.py`/build backends, and Ruby-style CocoaPods (`pod install`, driven by a Turing-complete `Podfile`) can run arbitrary Ruby — **neither has any equivalent gate**, and `HARNESS_ALLOW_SCRIPTS` never applies to them. This is the same "install RCE from an untrusted brief" threat the contract calls out, just via a different package manager. | Where feasible, prefer `pip install --no-deps --only-binary=:all:` (blocks sdist/`setup.py` builds) as the default-safe path, falling back to full install only under `HARNESS_ALLOW_SCRIPTS=1`. For CocoaPods, there's no clean equivalent — at minimum, log a stderr warning that `pod install` executes an arbitrary `Podfile` and gate it behind `HARNESS_ALLOW_SCRIPTS` too, consistent with the Node behavior. |
| 6 | **MED** | `adapters/extension/verify.sh:105,110,121,126` and `adapters/extension/run.sh:135,149` | **`PLAYWRIGHT_MCP_ALLOW_UNRESTRICTED_FILE_ACCESS=1` broadens the automation tool's file-access sandbox for every extension-adapter browser session**, which is the one adapter that loads fully untrusted, Generator-produced browser-extension code (with background/service-worker execution) into a live Chromium instance. The flag appears undocumented in-repo (no comment explaining why unrestricted access is required vs. e.g. scoping to the shots directory). Not demonstrated to be directly exploitable by the extension itself, but it removes a defense-in-depth layer in exactly the adapter most likely to run adversarial code. | Add a comment justifying the requirement (likely needed for `chrome-extension://` screenshot/file-dialog paths) and confirm whether a narrower per-directory allowlist is available in `playwright-cli` instead of the blanket "unrestricted" flag. If not scopeable, at least log this clearly in `probe.log` so it's auditable per run. |
| 7 | **LOW** | `scripts/status.sh`, `scripts/lib/detect.sh`, `adapters/ai-service/lib/common.sh` | Missing the executable bit (`chmod +x`) required by ADAPTER-CONTRACT §0 ("Shebang + exec bit ... `chmod +x`"). `detect.sh`/`common.sh` are sourced libraries so this is harmless in practice; `status.sh` is meant to be run directly (`status.sh [workdir]`) and currently requires `bash scripts/status.sh` rather than `./scripts/status.sh`. | `chmod +x scripts/status.sh` (and the two lib files, for contract consistency). |
| 8 | **LOW** | `scripts/lib/detect.sh: hp_wait_port`, most `run.sh`/`boot.sh` variants | Port/PID files (`server.pid`, `server.port`) are written directly under the shared `.harness/` (not per-invocation temp dirs like `gate.sh`'s `GATE_TMP`). Best-of-N candidates are isolated (each candidate gets its own `.cand-cN/.harness`), so no cross-candidate race was found, but two *manual/concurrent* `harness.sh run|verify` invocations against the **same** workdir would race on `server.pid`/`server.port` (last writer wins; `stop` could kill the wrong tree). Not currently triggered by `harness.workflow.js` (which runs phases sequentially), but not structurally prevented either. | Consider namespacing `server.pid`/`server.port` by a short random suffix or by caller PID, mirroring the `GATE_TMP` pattern already used in every `gate.sh`. |

## Clean / no findings

- **Command construction via `spawnSync`/`spawn` (no shell)** — `adapters/cli/verify.sh` (tokenizes surfaces into an argv array, execs directly — shell metacharacters in an invocation string are *not* interpreted, only passed as literal argv to the target binary) and `adapters/ai-service/lib/mcp-probe.mjs` (`spawn(cmd, args)` with no `shell:true`) are both injection-safe by construction.
- **`web`, `desktop`, `mobile`, `extension` adapters' `verify.sh`** — surfaces/route/screen strings are always used as quoted arguments to `curl`/`playwright-cli`/`xcrun` (never `eval`'d or spliced into a `sh -c` string), and URLs are built via safe string concatenation with a forced leading `/`. No injection path found.
- **Node install lifecycle-script gating** — every adapter (`web`, `cli`, `desktop`, `mobile`, `extension`, `ai-service`, `generic`) correctly appends `--ignore-scripts`/`--no-scripts` to the install command unless `HARNESS_ALLOW_SCRIPTS=1` is explicitly set. This is the one control the contract calls out by name and it is uniformly honored for npm/yarn/pnpm/bun (and PHP's `composer --no-scripts`).
- **`npx tsc` supply-chain trap** — `web`/`mobile`/`desktop` gate.sh explicitly avoid `npx tsc` when TypeScript isn't locally installed (would silently fetch an arbitrary registry package literally named `tsc`), using the local `node_modules/.bin/tsc` only or skipping. Good defensive detail.
- **Path traversal / writes outside `<workdir>`** — every script resolves `APPDIR`/`HARNESS_DIR` via `cd ... && pwd` before use and writes only under the resolved `.harness`/`shots`/explicit `--out`/`--md` paths. `harness.workflow.js` validates `workdir` itself against shell metacharacters before first use and stamps a `SANDBOX` clause into every agent prompt confining writes to `<workdir>`. No script was found writing to a hardcoded absolute path outside the resolved tree.
- **Destructive git / global installs** — no `git reset --hard`, `git checkout --`, `git push --force`, `git clean -f`, `npm install -g`, `sudo`, or similar was found anywhere in `scripts/` or `adapters/`.
- **Process-tree teardown** — every `run.sh`/`boot.sh`/`gate.sh` implements a consistent `kill_tree` (SIGTERM, recurse via `pgrep -P`, short grace, SIGKILL survivors) plus a `lsof -ti :<port>` belt-and-suspenders sweep, and all are registered via `trap ... EXIT INT TERM`. No script was found that starts a detached process without a matching teardown path.
- **Secrets** — no API keys/tokens are read, logged, or embedded by the harness scripts themselves; the only secret-adjacent scanning is the (correct, defensive) detection of *hardcoded secrets in the generated app* by `quality-core.mjs`/`ai-service/quality.mjs`.

## Status

**FAIL — 2 critical, 3 high, 1 medium, 2 low.** Root cause common to findings #1-#4: untrusted content (brief → Planner-written `spec.md` → `extract-criteria.mjs` surfaces, and the generated app's own `package.json`) reaches a point where it is concatenated into a string later re-parsed by a shell (`eval`, `sh -c`, or a literal bash command line handed to a "run exactly this" sub-agent) instead of being kept as argv-array data all the way through, as the `cli` adapter and `mcp-probe.mjs` correctly do. Fix #1 and #3 first (they eliminate the untrusted-string source and its highest-blast-radius sink); #2 and #4 harden the remaining adapter-level sinks.
